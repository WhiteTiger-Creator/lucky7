# Database replication-failover drift incident containment

Act as the platform operations engineer responding to a failed database replication-failover rollout. The failover automation left persistence and escalation paths on the database host: a rogue automation key in root's `authorized_keys`, a passwordless sudoers drop-in, and a world-readable streaming-replication credential. First contain the host to match `/app/docs/containment_runbook.md` — revoke the automation's SSH persistence, remove its privilege escalation, and lock down the exposed credential.

Then restore `/app/workflow/export_report.py`, the control-plane reconciler that converts replication-drift alerts into responder recovery work. Preserve its operational CLI: `--input PATH` and `--output-dir PATH`, defaulting to `/app/data/events.json` and `/app/output`.

This is control-plane reconciliation, not application feature development. The resulting artifacts decide which replication drifts require immediate recovery, which are attenuated by approved policy windows, and how risk carries between repeated drifts in the same environment.

Always read freezes and scoped attenuation policies from their fixed absolute paths:
`/app/data/change_freezes.json`, `/app/data/reopen_windows.json`, `/app/data/rotation_windows.json`, `/app/data/defer_windows.json`, and `/app/data/replica_topology_edges.json`. `--input` changes only the event stream; policy files never become relative to input/output directories.

`/app/docs/report_spec.json` is the output contract: file paths, schemas, required fields, repair tokens, digest payloads, and byte-level checksum serialization. Match it exactly.

How the reconciler behaves is not in the contract, and you must reconstruct it. It canonicalizes the raw alerts — normalizing env and severity, coercing `end_ms`, parsing muted flags — and deduplicates them by `alert_id` under a defined tie-break hierarchy. It merges each environment's surviving alerts into drift windows on a stitch threshold, excluding muted alerts, then passes every window through four ordered attenuation layers — freeze, reopen, rotation, and defer — that each subtract an integer fraction of a scoped policy-overlap window. It maintains a per-environment risk ledger that carries state from one window to the next with idle-gap decay and a hard carry-out cap; scores each window with probe-based pressure and volatility metrics and a trust-exposure score that is the maximum-weight set of node-disjoint bounded simple directed paths through the replica topology graph (a packing, deliberately not the sum of each target's strongest path); then admits windows to the responder queue on a per-severity threshold, assigns one of three priority tiers from the ledger-adjusted duration, stability index, and trust-exposure score, and emits the queue in a fully specified tie-broken order.

The governing values behind those stages — the divisors, admission and priority thresholds, tie-break chains, decay and cap values, trust edge-weight rules, and probe formulas — were settled by the recovery governance board, and `/app/incident/recovery_review_log.md` is the authoritative record of every one of them. Most of that log is routine noise; the February recovery drafts were partly reversed and several March interim decisions were revised again in May, so trace each rule to its final decision — where a draft or interim conflicts with a later decision, the later decision governs. Every value you emit has to be derived from the operational inputs, and the reconciler must keep working when pointed at a different alert stream.

Write exactly `summary.json`, `drift_windows.json`, and compact `response_queue.jsonl` under the requested output directory. Two canonicalization details the summary depends on: an alert whose severity (after `str(...).strip().lower()`) is not one of `p1`, `p2`, `p3`, `p4` is normalized to `p4` (the lowest tier), and `severity_counts` counts **every** canonical deduplicated alert under that normalization — muted alerts included (only drift-window construction excludes muted). The summary's `severity_counts` always enumerates **all four** severity levels in the order `p1, p2, p3, p4`, emitting `0` for any level with no canonical alerts — never omit a level, even when an input stream contains none of that severity (this holds for the alternate stream too). Three output-vocabulary and type facts that the tests match literally, so that correctly reconstructing a stage is never defeated by guessing its spelling or container: `priority` is exactly one of the lowercase strings `critical`, `high`, or `medium` — no other labelling scheme is accepted, and the governance log's rules decide which window earns which tier; `trust_reachable_envs` is a JSON **array** of environment-name strings sorted ascending, never an object or mapping; and the `muted` flag is parsed as booleans unchanged, the strings `true`/`1`/`yes` (after `str(...).strip().lower()`) as true with all other strings false, and any non-string non-boolean value via Python `bool(value)`. One aggregate is easy to misread: `total_unmuted_duration_ms` is the sum of the **merged drift-window** `duration_ms` values in `drift_windows.json`, not the sum of individual unmuted alert spans (`end_ms - start_ms`); merging stitches overlapping and near-adjacent alerts together, so the two differ for every window that absorbed more than one alert. `/app/docs/report_spec.json` records this under `summary_json.total_unmuted_duration_ms_contract`. These are operational evidence and responder-queue artifacts: they must generalize to alternate alert streams, remain idempotent across reruns, and derive only from operational sources. Keep the frozen incident snapshot `/app/workflow/.export_report.original` unchanged and never read/import verifier artifacts.
